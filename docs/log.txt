# Start from DeepStream 7.0 Triton container
FROM nvcr.io/nvidia/deepstream-l4t:7.0-triton-multiarch

# Install dependencies
RUN apt-get update && apt-get install -y \
    wget curl python3-pip unzip \
    && rm -rf /var/lib/apt/lists/*

# Install NGC CLI (ARM64 version 4.5.2)
RUN wget --content-disposition \
    https://api.ngc.nvidia.com/v2/resources/nvidia/ngc-apps/ngc_cli/versions/4.5.2/files/ngccli_arm64.zip \
    -O ngccli_arm64.zip && \
    unzip ngccli_arm64.zip && \
    rm ngccli_arm64.zip && \
    chmod u+x ngc-cli/ngc && \
    mv ngc-cli /usr/local/bin/ && \
    ln -s /usr/local/bin/ngc-cli/ngc /usr/local/bin/ngc

# Install TAO Toolkit Converter for Jetson (JetPack 6.0, ARM64)
RUN ngc registry resource download-version nvidia/tao/tao-converter:v5.1.0_jp6.0_aarch64 && \
    cd /opt/nvidia/deepstream/deepstream-7.0/tao-converter_vv5.1.0_jp6.0_aarch64 && \
    chmod +x tao-converter && \
    mv tao-converter /usr/local/bin/ && \
    cd / && rm -rf /opt/nvidia/deepstream/deepstream-7.0/tao-converter_vv5.1.0_jp6.0_aarch64

# Set default workdir
WORKDIR /workspace

- first attempt to start the recorder failed: "Failed to create CaptureSession" from gst-nvarguscamerasrc.
  > issue: nvargus-daemon got into a stuck state from a previous run. journal showed "NvPclInitializeDrivers: error: Failed to init camera sub module v4l2_sensor" and "NvPclStateControllerOpen: Failed ImagerGUID 0".
  > solution: it cleared on its own after a moment. if it ever sticks again: sudo systemctl restart nvargus-daemon. a plain gst-launch-1.0 nvarguscamerasrc ! ... ! fakesink is a good quick probe to confirm the camera is healthy.


- recorded a clip with the app but the playback was ~2x sped up.
  > issue: tools/record.py was using cv2.VideoWriter(..., fourcc="mp4v", fps=cfg.framerate=30). the gui tick loop does cam.read + writer.write + PIL+Tk preview render every frame, which on orin nano caps the loop at ~15 fps. file was tagged 30fps so playback runs ~2x.
  > confirmed with cv2.CAP_PROP_FRAME_COUNT / CAP_PROP_FPS vs file mtime - filename minus warmup: 1499 frames tagged 30fps gave "49.97s" but real wall-clock was ~100s.


- tried switching to a GStreamer pipeline writer with nvv4l2h264enc (hardware encode).
  > issue: nvv4l2h264enc does not exist on jetson orin nano. the silicon block (NVENC) is not present — this is a hard platform constraint. /usr/lib/aarch64-linux-gnu/gstreamer-1.0/libgstnvvideo4linux2.so is installed but only exposes nvv4l2decoder.
  > verify with: gst-inspect-1.0 /usr/lib/aarch64-linux-gnu/gstreamer-1.0/libgstnvvideo4linux2.so
  > solution: use x264enc (software). orin nano has 6x A78AE cores, x264enc speed-preset=superfast tune=zerolatency threads=4 sustains realtime at 720p.


- writer.write() failed with "Error pushing buffer to GStreamer pipeline" / not-negotiated.
  > issue: i had put video/x-raw,format=BGR,width=W,height=H,framerate=N/D right after appsrc. cv2.VideoWriter with CAP_GSTREAMER silently overrides the appsrc framerate caps with its own integer fraction computed from the fps argument. when my caps filter declared a different framerate, downstream pre_eventfunc_check refused the caps mismatch.
  > GST_DEBUG=2 showed: "caps video/x-raw, format=(string)BGR, ... framerate=(fraction)3014/95 not accepted" at capsfilter3:sink.
  > solution: do not declare framerate in any caps filter between appsrc and videoconvert. let opencv's framerate flow through.


- writer worked but file played back ~6% fast even with the GStreamer pipeline.
  > issue: my fps warmup was a blocking loop in _start_recording (cam.read + probe.write only, no GUI render). measured ~31 fps. but during real recording the gui tick loop also has PIL+Tk render, dropping actual to ~18 fps. declared > actual = sped up.
  > solution: moved the warmup INTO _tick so it runs through the same code path as real recording (camera read + encode probe + PIL+Tk render). warmup state is now (probe_writer + counter + start_time). when 1.5s of frames have accumulated, probe is released, fps is computed, real writer opens. measured fps now matches actual recording fps.
  > pipeline used: appsrc is-live=true format=time ! queue ! videoconvert ! video/x-raw,format=I420 ! x264enc speed-preset=superfast tune=zerolatency threads=4 key-int-max=2*fps bitrate=8000 ! h264parse ! mp4mux faststart=true ! filesink location=...


- side note: appsrc do-timestamp=true is ALSO a no-op with opencv's cv2.VideoWriter for the same reason — opencv pre-stamps each buffer's PTS from the declared fps before pushing to appsrc, so the running-time stamp from do-timestamp never gets applied. tested by declaring fps=120 but pushing at 20 fps over 2s — file duration was 0.33s (40/120) not 2s.
